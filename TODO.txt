## 当前需要重点重构的问题

用户发现：只靠 query 检索召回原始 chunk，对“总结论文、总结方法、生成中文 Word 和英文 LaTeX 报告”这类任务不友好。

曾经加入按论文章节召回的 tool，但问题是：

- 如果直接返回整章内容，token 消耗太大。
- 如果 `get_section_summary` 在用户请求时现场调用 LLM 总结整章，本质上仍然没有解决 token 问题，只是把成本从 retrieve chunks 转移到 summary tool。

生产级设计结论：

- 总结类工具应该读取“预先生成并缓存的结构化摘要”。
- 原始 chunk 检索工具只用于证据查找、引用和核验。
- 长耗时摘要生成应该任务化、缓存化、可重试、可观测。

## 下一阶段目标工具

新增或重构以下 MCP tools：

### 1. `get_paper_outline`

用途：快速了解论文结构，不返回正文。

返回建议：

```json
{
  "paper_id": "local_xxx",
  "title": "...",
  "sections": [
    {
      "section": "Abstract",
      "chunk_count": 2,
      "char_count": 3200,
      "has_summary": true
    },
    {
      "section": "Method",
      "chunk_count": 8,
      "char_count": 16800,
      "has_summary": false
    }
  ]
}
```

### 2. `get_paper_profile`

用途：给 LLM 一个短上下文，判断论文是否相关，适合任务规划和多论文比较前的筛选。

返回应高度压缩：

```json
{
  "paper_id": "local_xxx",
  "title": "...",
  "problem": "...",
  "background": "...",
  "method": "...",
  "contributions": ["...", "..."],
  "experiments": "...",
  "limitations": "...",
  "keywords": ["SCMA", "MPA", "Log-MPA"]
}
```

建议控制在 1000 到 2000 中文字符，或 600 到 1200 英文 tokens。

### 3. `get_paper_summary`

用途：面向“总结论文”“总结方法”“生成报告初稿”等任务。

参数建议：

```json
{
  "paper_id": "local_xxx",
  "detail_level": "short | medium | long",
  "language": "zh | en",
  "include_sections": true,
  "include_evidence": false,
  "max_chars": 12000
}
```

返回建议：

```json
{
  "paper_id": "local_xxx",
  "summary_status": "ready",
  "detail_level": "medium",
  "summary": {
    "background": "...",
    "problem": "...",
    "method": "...",
    "experiments": "...",
    "results": "...",
    "limitations": "...",
    "takeaways": "..."
  },
  "section_summaries": [
    {
      "section": "Method",
      "summary": "..."
    }
  ],
  "truncated": false
}
```

如果摘要未生成，不应现场长时间总结，应该返回结构化错误：

```json
{
  "status": "not_ready",
  "error_code": "SUMMARY_NOT_READY",
  "message": "Paper summary has not been generated yet.",
  "suggested_tool": "build_paper_summary",
  "paper_id": "local_xxx"
}
```

### 4. `retrieve_evidence_chunks`

用途：重构原有 `retrieve_local_papers`，只服务于证据查找、引用、核验，不承担整篇总结。

参数建议：

```json
{
  "query": "message passing approximation method",
  "paper_ids": ["local_xxx"],
  "section": "Method",
  "top_k": 5,
  "max_chars": 6000
}
```

返回必须包含：

```json
{
  "chunks": [
    {
      "paper_id": "local_xxx",
      "title": "...",
      "section": "Method",
      "chunk_id": "...",
      "score": 0.82,
      "text": "...",
      "page_start": 3,
      "page_end": 4
    }
  ],
  "truncated": true,
  "next_cursor": "..."
}
```

tool schema 描述里要明确：

- 用于 exact supporting passages、citations、evidence。
- 不应作为 general paper summarization 的首选工具。

### 5. `build_paper_summary`

用途：后台构建摘要缓存。可以显式调用，也可以在论文导入后自动创建任务。

参数建议：

```json
{
  "paper_id": "local_xxx",
  "language": "zh | en | both",
  "detail_levels": ["short", "medium", "long"],
  "force_rebuild": false
}
```

返回 job id，使用现有 `get_tool_job_status` 查询进度。

## 推荐摘要生成策略

不要一次把整篇论文丢给模型。

推荐 Map-Reduce 风格：

1. chunk 级压缩
   - 每个 chunk 提取极短要点。
   - 由配置的 summary model 完成；模型不可用时任务失败并返回结构化错误。
2. section 级摘要
   - 基于该章节的 chunk 要点生成章节摘要。
3. paper profile
   - 基于章节摘要生成固定字段画像。
4. paper summary
   - 基于章节摘要生成 short / medium / long 三档摘要。

这样 token 成本发生在导入后或后台任务阶段，用户后续多次查询直接读取缓存。

## 本地总结模型建议

默认推荐：`Qwen/Qwen3-8B`

原因：

- 英文论文总结够用。
- 中文输出也稳定，适合用户需要中文 Word 总结和英文 LaTeX 总结的双语场景。
- 模型规模适合本地部署。
- vLLM 支持 Qwen3。
- 原生上下文 32K，通常足够做 section-level summary。

不要因为模型支持长上下文就整篇论文一次性塞入模型。仍应按 chunk -> section -> profile/summary 分层生成。

建议配置：

```env
RAG_SUMMARY_MODEL_NAME=Qwen/Qwen3-8B
RAG_SUMMARY_BACKEND=vllm
RAG_SUMMARY_MODEL_PATH=rag/models/summary-model
RAG_SUMMARY_DEVICE=cuda
RAG_SUMMARY_GPU_MEMORY_UTILIZATION=0.90
RAG_SUMMARY_CPU_OFFLOAD_GB=0
RAG_SUMMARY_MAX_INPUT_TOKENS=4096
RAG_SUMMARY_MAX_OUTPUT_TOKENS=2048
RAG_SUMMARY_TEMPERATURE=0.2
RAG_SUMMARY_TOP_P=0.8
RAG_SUMMARY_CONCURRENCY=1
RAG_SUMMARY_TIMEOUT_SECONDS=300
RAG_SUMMARY_ENFORCE_EAGER=true
```

硬件分档建议：

- 8GB 显存：`Qwen3-4B` 或 `gemma-3-4b-it` 量化，只做 chunk/section 摘要。
- 12GB 显存：`Qwen3-8B` INT4/AWQ/GPTQ。
- 16GB 显存：`Qwen3-8B` 量化，推荐主力配置。
- 24GB 显存：`Qwen3-14B` 量化，或 `gemma-3-12b-it`，用于更高质量离线摘要。
- 48GB+ 显存：`Qwen3-32B` 量化，适合高质量离线摘要，不建议默认在线使用。

## 生产级注意事项

### 1. 异步任务不能阻塞 MCP

- `add_paper_to_database` 不应等待本地 LLM 完成总结。
- 导入后可以创建 summary job。
- 长任务统一通过 job id 查询。
- MCP stdio tool 必须快速返回，避免 Claude Code/Codex 调用后长时间无响应。

### 2. 所有摘要必须可追溯

每条 summary 都要记录：

- paper_id
- section_id 或 section_name
- source_hash
- model_name
- prompt_version
- generated_at
- language
- detail_level
- status
- error_code

### 3. 必须支持重新生成

需要支持：

- `force_rebuild=false`
- `language`
- `detail_level`
- `summary_version` 或 `prompt_version`

### 4. 摘要不能替代证据

总结类工具返回的是压缩后的理解，适合报告初稿、方法概述和快速浏览。

如果用户需要引用、核验、逐句依据或比较细节，仍应调用 `retrieve_evidence_chunks` 获取原文片段。

### 5. 控制 token 和输出大小

所有内容返回工具都应支持：

- `max_chars`
- `detail_level`
- `include_evidence`
- `truncated`
- `next_cursor`

### 6. 失败必须结构化

建议错误码：

- `SUMMARY_NOT_READY`
- `SUMMARY_MODEL_UNAVAILABLE`
- `SUMMARY_GENERATION_TIMEOUT`
- `SUMMARY_GENERATION_FAILED`
- `PAPER_NOT_FOUND`
- `SECTION_NOT_FOUND`
- `TOKEN_BUDGET_EXCEEDED`
- `VECTOR_STORE_UNAVAILABLE`
- `INVALID_ARGUMENT`

### 7. 本地模型加载要独立管理

不要每次 tool 调用都加载模型。

需要：

- summary model manager
- 健康检查
- 显存检查
- 并发限制
- 超时控制
- 懒加载或启动预热

### 8. 摘要质量需要评估

evaluation 中应增加 summary 评估数据：

- profile 字段是否完整
- method 是否覆盖核心步骤
- contribution 是否准确
- experiments/results 是否保真
- 是否有幻觉
- 是否能支撑中文 Word 和英文 LaTeX 报告生成

### 9. 日志不能记录大段论文正文

日志只记录：

- paper_id
- job_id
- stage
- elapsed_ms
- model_name
- token_count
- error_code

不要记录：

- 大段 chunk 正文
- 完整 prompt
- 完整摘要正文
- 用户本地绝对敏感路径

### 10. 安全边界

- PDF 路径限制在配置的 papers 目录内。
- 模型路径限制在配置的 models 目录内。
- evaluation/finetune 输入输出路径限制在项目允许目录内。
- 不允许 tool 读取任意本地路径。
- 不允许用户通过参数传任意 prompt 模板路径。

## 当前最重要的工程判断

`get_section_summary`、`get_paper_summary`、`get_paper_profile` 不能设计成实时读取整篇论文并调用 LLM 的工具。它们应该是读取预计算缓存的轻量工具。

真正耗时、耗 token 的摘要生成应发生在导入后的后台任务或显式 `build_paper_summary` 任务中，并且结果要可缓存、可复用、可追踪、可重建。
